# wight框架

wight诞生于2020年7月，当时居家办公，闲暇时间开始构思一个用于开发前端应用的框架，主要目的是为了方便自己快速开发，而现在仍然是主要针对自己快速开发而设计，也包括自己的团队使用，我是不打算做什么推广的，只是希望它就是一个十分小众的框架。

vue在2.5的时候尝试过开发一些应用，因为一些问题而放弃，后来使用过一个比较小众的框架：mithril.js。也是因为种种问题都放弃了。再后来计划使用react，但是当时搭建环境竟然就要1000多个扩展的依赖，依赖越多，不稳定性越高，关键是这仅仅是一个前端环境，是一个不需要特别繁重的开发环境，但是需求变化却很剧烈的领域。

总之，各种因素加上我本身喜欢自己创造一些东西，并且我需要以下需求：

- 扩展依赖尽可能少。

- 抛弃虚拟DOM。

- 页面可以保留，像系统的一个进程一样，可以保留栈环境。

- 也要支持模块化和打包。

- 可以同时启动多个项目，而不是一个服务只能运行一个。

- 可以在开发环境和实际部署尽可能实现无缝衔接，打包后的文件直接给后台服务即可部署。

- 支持自定义组件，并且开发过程中要检测到循环依赖然后提示错误。

- 尽可能使用浏览器原生功能或对原生功能做更方便使用的封装。

- 鼓励探索并尽可能使用原生API。

- 通过一个配置文件用于描述项目的各项配置，并通过配置即可实现项目的基本信息以及打包的过程控制。

- 开发过程中可以自动检测到文件保存并自动重构项目，前端要自动刷新，前端的错误前端提示，后台的js文件出错后台报错。

- 若开发中，实时构建项目出现错误则只有后台报错。没有错误，前端才会自动刷新。

因此就开发了此框架，命名为wight，因为当初它确实很怪异，至少目前来看都是一个不太寻常而框架。这些需求也不是一次解决的，是经历过几次大的更新，并且当时经历过一次整体重新的设计。

wight尽可能使用原生浏览器功能，摒弃虚拟DOM，鼓励在必要的时候使用DOM的API。因此带来的好处是一切前端应用不必因为虚拟DOM而做优化甚至不得不放弃框架采用其他方案。

> 比如，你要做实时渲染以及canvas和前端游戏类的应用。
> 或者，你需要更加细致的控制某个元素，这在通常的虚拟DOM框架实现起来相对很复杂。

请不要考虑在html中使用模板代码，这种方式早期版本支持，现在已经废弃。

所有的逻辑操作都在js文件中完成，并通过模板字符串(字符串字面量)完成页面的显示。

html文件完成固定的结构，并通过data-name来确定一个可以被替换的容器，通过data-map确定数据渲染的方法：

```html
<div data-name=info data-map=renderinfo></div>
```

在js文件中，通过this.view来渲染到页面：

```javascript
this.view({
    info: {name: 'OO', id: '100293', detail: 'ok, success'}
})
```

那应该以哪种形式来渲染到html中，这需要在页面或组件的js文件中提供rendinfo方法，框架层面会把数据传递到renderinfo并把返回的结果渲染到页面上：

```javascript
renderinfo(ctx) {
    //ctx.data是数据，ctx.target是具体的DOM节点。
    let d = ctx.data
    return `<h4>${d.name}</h4>
        <div row>
            <div cell>${d.id}</div>
            <div cell>${d.detail}</div>
        </div>`
}
```

如果采用模板字符串，并支持简单的表达式，这就会导致大量的计算逻辑放在了html文件中，因此把逻辑和渲染做了彻底的分离。html中不会出现计算的东西。

注意：以后也不会考虑支持JSX，也就是说js中拼接html就是模板字符串，并直接进行渲染(不必进行转码)，而不是react采用的JSX。

**如果渲染时html代码出现错误，会进行提示，不会因为标签引用错误导致整个页面不正常。**
